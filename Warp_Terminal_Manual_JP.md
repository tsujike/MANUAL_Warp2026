## 1. Warpとは何か

Warp 2.0 は単なるターミナルエミュレータではなく、**ADE（Agentic Development Environment）** と位置づけられている。4つの柱で構成されている。

| 柱 | 役割 |
|---|---|
| **Terminal** | 従来のコマンドライン操作。ブロック単位の出力表示やモダンなテキスト編集機能 |
| **Code** | 組み込みのコードエディタ。ターミナルと同一画面でファイルを編集 |
| **Agents** | AIエージェント。自然言語でコマンドを実行・デバッグ・コード生成 |
| **Drive** | Warp Drive。ワークフロー、ノートブック、ルール、環境変数を保存・共有 |

Windows版ではPowerShell、WSL（Windows Subsystem for Linux）、Git Bashのいずれかをシェルとして選択できる。

## 2. ワークスペースとしての使い方

### 2.1 基本的な画面構成

Warpを開くと、1つのウィンドウの中にタブとペインがある。タブはブラウザのタブと同じ概念で、ペインは1つのタブ内の分割画面。

```
┌──────────────────────────────────────────┐
│ [タブ1] [タブ2] [タブ3]                     │
├──────────────────┬───────────────────────┤
│                  │                       │
│   ペイン 1        │     ペイン 2           │
│  (PowerShell)    │    (WSL / bash)       │
│                  │                       │
├──────────────────┴───────────────────────┤
│                                          │
│   ペイン 3（下段全幅）                      │
│                                          │
└──────────────────────────────────────────┘
```

### 2.2 Launch Configurations（起動設定）

プロジェクトごとにウィンドウ・タブ・ペインの構成を保存しておき、名前で呼び出して再現できる機能。YAMLファイルとして裏で保存されるが、普段はCommand Paletteやメニューから名前で呼び出すだけでよい。

**保存方法（現在のレイアウトを保存）:**

1. まず使いたいレイアウト（タブ数、ペイン分割、作業ディレクトリ）を手動で作る
2. Command Palette を開く（`Ctrl+Shift+P`）
3. 「Save New Launch Configuration」と入力
4. 名前をつけて保存

**呼び出し方法（保存した設定を開く）:**

| 方法 | 操作 |
|---|---|
| Command Palette から | `Ctrl+Shift+P` → 「Launch Configuration」と入力 → 一覧から選択 |
| タブの「+」ボタン右クリック | 表示されるメニューから保存済みの設定を選択 |

**上級者向け: YAMLファイルを手動で作成・編集する場合:**

保存先: `~/.warp/launch_configurations/`

```yaml
# 例: my-project.yaml
name: "My Web Project"
windows:
  - tabs:
      - title: "Frontend"
        layout:
          panes:
            - command: "cd ~/projects/frontend && npm run dev"
            - command: "cd ~/projects/frontend && code ."
          split_direction: horizontal
      - title: "Backend"
        layout:
          panes:
            - command: "cd ~/projects/backend && python manage.py runserver"
      - title: "Database"
        layout:
          panes:
            - command: "psql -U myuser -d mydb"
```

UIから保存した設定もこのフォルダにYAMLとして保存されるので、後からテキストエディタで細かく調整することもできる。

### 2.3 Warpの概念整理 — VSCodeの「Workspace」に相当するものは？

Warpには VSCode のような「Workspace」という単一の機能はない。似た概念が複数の機能に分散している。

| Warpの概念 | VSCodeで言うと | 何をするもの |
|---|---|---|
| **Launch Configuration** | ウィンドウレイアウトのテンプレート | タブ・ペインの配置と起動コマンドを保存・再現する |
| **Session** | 開いているウィンドウそのもの | 今動いているターミナルの実体（ランタイム） |
| **Project** | フォルダを開いた状態 | Gitリポジトリを開くと自動認識。コードインデックスや AGENTS.md が紐づく |
| **File Tree** | エクスプローラー（サイドバー） | プロジェクトのファイル構造をツリー表示。ファイルをクリックで内蔵エディタに開く |
| **Codebase Context** | ― (VSCodeにはない) | インデックス済みの複数リポジトリをエージェントが横断参照できる仕組み |
| **Warp Drive** | ワークスペース設定 + 共有フォルダ | Workflow、Notebook、Rules、環境変数を保存・共有する場所 |
| **Conversation** | ― (VSCodeにはない) | エージェントとの会話スレッド。1タブ内に複数持てる |

**VSCodeの「フォルダを開く」に一番近い操作:**

Warpでは、タブにプロジェクト（Gitリポジトリ）を**開く**ことができる。タブにプロジェクトを開くと以下が起こる:

1. そのプロジェクトのディレクトリが作業ディレクトリになる
2. 左サイドバーの **File Tree** にプロジェクト構造がツリー表示される
3. Warpがコードベースの **インデックス**（初回時は `/init` フロー）を開始する
4. `AGENTS.md` があればエージェントがルールとして読み込む

**Launch Configuration はその「プロジェクトを開いたタブ」のレイアウトをまるごと保存・再現する**テンプレートという位置づけ。

```
Launch Configuration を開く
    │
    ├── ウィンドウ・タブ・ペインの形が再現される
    │
    └── 各タブでプロジェクトが開かれる
            │
            ├── File Tree にプロジェクト構造が表示
            ├── コードベースのインデックス（Codebase Context）
            └── AGENTS.md を読み込み
                    │
                    └── エージェントがプロジェクト固有の文脈を理解
```

### 2.4 プロジェクト管理と Codebase Context

#### インデックスの仕組み

Gitリポジトリを開くと、Warpが自動的にプロジェクトとして認識し、コードベースのインデックスを開始する。**手動での操作は不要**で、バックグラウンドで勝手にやってくれる。

**重要: インデックスはタブを閉じても消えない。** インデックスはタブやセッションとは独立してWarpの設定に永続保存される。一度インデックスが完了すれば、タブを閉じてもWarpを終了しても、Settings > Code > Codebase Index にそのまま残り続ける。つまり「タブで開く」のはインデックス登録のきっかけに過ぎず、開きっぱなしにしておく必要はない。

インデックスの同期タイミング:
- 初回のインデックス登録時
- 定期的な自動同期
- 新しいエージェント会話を開始したとき
- Settings > Code で手動で同期ボタン（🔄）を押したとき

#### インデックス関連の設定（Settings > Code）

`Ctrl+,` で Settings を開き、**Code** セクションに移動すると、インデックスに関する設定が2つある。

| 設定項目 | 役割 |
|---|---|
| **Codebase Indexing** | インデックス機能自体のオン/オフ。オフにするとインデックス機能が完全に無効になる。**オン推奨** |
| **Index new folders by default** | 新しいGitリポジトリに入ったとき、自動でインデックス対象に追加するかどうか。**状況による（下記参照）** |

**「Index new folders by default」をオンにすべきか？**

一見オンにしておくと楽だが、**注意点がある**:
- 新しいリポジトリを開くたびにインデックスが追加され、大きなリポジトリの場合は**インデックス中にCPUとメモリが一時的に跳ね上がる**
- インデックス済みリポジトリが増え続けると管理が煩雑になる
- プランごとにインデックス対象の**ファイル数上限**がある

**推奨: プロジェクト数が多い場合はオフのまま、必要なリポジトリだけ手動で管理する方が安定する。** エージェントに参照させたいリポジトリだけをSettings > Code > Codebase Indexで管理すれば、不要な負荷を避けつつクロスリポジトリ参照の恩恵を受けられる。

逆にプロジェクト数が少なく、全部エージェントに見せたい場合はオンでも問題ない。

#### インデックスの負荷を軽減する `.warpindexingignore`

リポジトリ内に `.warpindexingignore` ファイルを作成すると、インデックス対象から特定のファイルやフォルダを除外できる。書式は `.gitignore` と同じ。

```
# .warpindexingignore の例
node_modules/
dist/
build/
*.min.js
*.map
vendor/
__pycache__/
.next/
```

**効果:**
- インデックスにかかる時間とリソース消費が減る
- エージェントが不要なファイルを参照しなくなり、回答の精度が上がる
- 除外されたファイルはプランのファイル数上限にカウントされない

#### `/init` コマンド（手動セットアップ）

自動インデックスが有効でない場合や、`AGENTS.md` の雛形を自動生成したい場合は、エージェントモードで `/init` と入力することで手動実行できる。

```
> /init
```

→ コードベースをスキャンしてインデックスを作成し、AGENTS.md の雛形を生成してくれる

#### インデックスの管理画面

インデックスの状態は **Settings > Code > Codebase Index** で確認・管理できる。

| ステータス | 意味 |
|---|---|
| **Synced** | インデックス完了。エージェントのコンテキストとして利用可能 |
| **Discovering files** | スキャン・インデックス中 |
| **Failed** | インデックス失敗。再試行が必要 |

**この画面でできること:**
- どのリポジトリがインデックス済みか一覧で確認
- 失敗したリポジトリの再インデックス
- 不要になったリポジトリのインデックスを削除
- 自動インデックスのオン/オフの切り替え

**重要: クロスリポジトリ参照**

一度インデックスしたリポジトリは、**現在そのリポジトリにいなくてもエージェントが参照できる**。エージェントはインデックス済みの全リポジトリのファイルパスを把握しているため、別のプロジェクトにいても他のリポジトリのコードを引っ張ってこられる。

**使い方（2パターン）:**

**① プロンプトにリポジトリ名を書く（自動参照）:**

エージェントへの指示の中でリポジトリ名を明示するだけで、エージェントが裏側で自動的にそのリポジトリを探しに行く。スラッシュコマンドなどで呼び出す必要はない。

```
> ContainerShippingManager2026のDockerfileの構成を参考にして、
  このプロジェクトにもDockerfileを作って
```

→ エージェントが自動でContainerShippingManager2026のインデックスを参照し、Dockerfileの内容を踏まえて作業する

**② `@` メンションでファイルを添付（明示的な指定）:**

入力欄で `@` を打つとファイル・フォルダの検索メニューが開き、選択したファイルをエージェントのコンテキストに添付できる。検索はGitリポジトリのルートからの相対パスで行われ、`.gitignore` で除外されたファイルは表示されない。

```
> @src/utils/auth.ts このファイルのバグを直して
```

**クロスリポジトリ参照のコツ:**
- 参照したいリポジトリは事前にインデックス済みであることが前提（Settings > Code > Codebase Index で確認）
- プロンプトにリポジトリの**正確な名前**を入れると、エージェントが参照する確率が上がる
- `@` メンションは主に今いるリポジトリ内のファイル参照向け。別リポジトリのファイルを指定したいときはプロンプトにリポジトリ名を書く方法が確実

#### 実践: 全社共通リポジトリをインデックスに入れておく

クロスリポジトリ参照の最も効果的な使い方は、**会社全体のルールや標準が入ったリポジトリを常にインデックス済みにしておく**こと。

例えば以下のような内容を含むリポジトリ:
- 会社の開発哲学・コーディング規約
- GitFlowの運用マニュアル
- ブランチ命名規則やコミットメッセージの書式
- CI/CDパイプラインのテンプレート
- コードレビューのチェックリスト

このリポジトリをインデックスに入れておけば、**どのプロジェクトで作業していても**エージェントが会社の標準を参照できるようになる。

```
> tgglobal-standardのGitFlowルールに従ってこの機能のブランチを切って
```

→ エージェントが全社標準リポジトリからGitFlowのルールを参照し、正しい命名規則でブランチを作成

```
> うちのコーディング規約に沿ってこのコードをリファクタリングして
```

→ エージェントが標準リポジトリのコーディング規約を参照して、規約に準拠した形にリファクタリング

**セットアップ手順:**
1. Warpで全社標準リポジトリを一度タブで開く（例: `D:\Warp_Repository\tgglobal-standard`）
2. Codebase Indexingがオンであれば自動でインデックスが開始される
3. Settings > Code > Codebase Index でステータスが「Synced」になれば準備完了
4. 以降は**どのプロジェクトにいてもエージェントがこのリポジトリを参照可能**

**ポイント:** 全社標準リポジトリは内容が比較的小さく安定しているため、インデックスの負荷も軽い。「Index new folders by default」をオフにしていても、この1つだけは手動でインデックスに入れておく価値が高い。

**標準リポジトリが更新されたとき:**

全社標準リポジトリに更新が入った場合、ローカルに `git pull` で最新を取得する必要がある。pullすればインデックスは自動で差分更新されるので、再インデックスの手動実行は不要。

ただし**pullは手動だと忘れがち**なので、自動化しておくのが望ましい。

**方法1: Warpの Launch Configuration に git pull を仕込む**

Warpを起動したときに自動で全社標準リポジトリの最新を取得するようにする:

```yaml
# launch_configurations/daily-work.yaml の一例
name: "Daily Work"
windows:
  - tabs:
      - title: "Standards Sync"
        layout:
          panes:
            - command: "cd D:/Warp_Repository/tgglobal-standard && git pull"
      - title: "作業用"
        layout:
          panes:
            - command: "cd D:/Warp_Repository/（自分のプロジェクト）"
```

→ Launch Configuration を開くたびに最初のタブで `git pull` が自動実行される

**方法2: Windows タスクスケジューラで定期的に pull**

Warpとは独立して、Windowsのタスクスケジューラに定期pullを登録しておく:

```bat
REM pull-tgglobal-standard.bat（バッチファイルとして保存）
cd /d D:\Warp_Repository\tgglobal-standard
git pull origin main
```

タスクスケジューラで「毎日始業時（例: 9:00）」や「ログオン時」にこのバッチを実行するよう設定すれば、意識せずとも常に最新の状態が保たれる。

**方法3: Git の post-merge フックや CI/CD 連携**

チーム側で Webhook を設定し、標準リポジトリに push があったら各メンバーのローカルに通知を飛ばす仕組みも可能。ただし大がかりになるので、まずは方法1か2で十分。

ただし大量のファイルが変わった場合はインデックス同期にラグが出ることがある。確実に最新のインデックスでエージェントを使いたいなら、**Settings > Code で同期ボタン（🔄）を手動で押す**のが一番確実。

### 2.5 File Tree（プロジェクトエクスプローラー）

左サイドバーの **Tools パネル** から File Tree タブを開くと、VSCodeのエクスプローラーと同じようにプロジェクトのファイル構造がツリー表示される。

**開き方:**
- 左サイドバーの Tools ボタンをクリック → File Tree タブを選択
- または `Ctrl+\` で左パネルを開き、File Tree タブを選択

**File Tree でできること:**
- ファイルをクリック → Warp内蔵のコードエディタで開く
- 右クリック → エージェントのコンテキストにファイルを添付
- 右クリック → ファイルパス（絶対/相対）をコピー
- ファイルの追加・削除・リネームはリアルタイムで反映される

## 3. ノートブック（Notebooks）の使い方

### 3.1 ノートブックとは

Warp Drive内にある **Notebooks** は、Markdownベースの「実行可能なドキュメント」。Jupyter Notebookのターミナル版と考えるとわかりやすい。

**通常のドキュメントとの違い:**

| 項目 | 通常のMarkdown | Warp Notebook |
|---|---|---|
| コマンド実行 | できない | クリック1つで実行可能 |
| リアルタイム同期 | なし | チーム全員にリアルタイム反映 |
| 引数パラメータ化 | なし | `{{変数名}}` で引数指定可能 |
| ワークフロー埋め込み | なし | 既存ワークフローの埋め込み可能 |
| オフライン利用 | ファイル依存 | 読み取り専用モードで利用可能 |

### 3.2 ノートブックの作成手順

1. **Warp Driveパネルを開く**: `Ctrl+Shift+\`
2. **新規作成**: 「+」アイコンをクリック → 「New Notebook」を選択
3. **またはCommand Paletteから**: `Ctrl+Shift+P` → 「Create New Notebook」と入力

### 3.3 ノートブックの書き方

ノートブックのエディタはMarkdown記法を認識する。

```markdown
# サーバーセットアップ手順

## 1. 依存パッケージのインストール

まずNode.jsの依存パッケージをインストールする。

​```bash
npm install
​```

## 2. 環境変数の設定

以下のコマンドで `.env` ファイルを作成する。
ポート番号は環境に合わせて変更すること。

​```bash
cp .env.example .env
echo "PORT={{ポート番号}}" >> .env
​```

## 3. 開発サーバーの起動

​```bash
npm run dev
​```
```

**ポイント:**
- コードブロック（` ``` `）内のコマンドは「Run」ボタンが表示され、クリックでターミナルに送信される
- `{{ポート番号}}` のような `{{二重波括弧}}` は実行時に入力を求められるパラメータになる
- 「+」ボタンで既存のWorkflowを埋め込むことも可能

### 3.4 ノートブックの共有

1. Warp Driveパネルで対象のノートブックを見つける
2. **Personal** フォルダからチームの **Shared** フォルダにドラッグ＆ドロップ
3. チームメンバー全員に編集権限が付与され、変更はリアルタイムに同期

### 3.5 エクスポート

ノートブックはMarkdown形式でエクスポートできるため、他のツールへの移行も容易。ベンダーロックインの心配はない。

## 4. タブとペインの切り替え・操作

### 4.1 タブ操作

| 操作 | ショートカット |
|---|---|
| 新しいタブを開く | `Ctrl+Shift+T` |
| タブを閉じる | `Ctrl+Shift+W` |
| 閉じたタブを復元 | `Ctrl+Alt+T` |
| 前のタブへ移動 | `Ctrl+PgUp` |
| 次のタブへ移動 | `Ctrl+PgDn` |
| タブ1〜8に直接ジャンプ | `Ctrl+1` 〜 `Ctrl+8` |
| 最後のタブにジャンプ | `Ctrl+9` |
| タブを左に移動 | `Ctrl+Shift+Left` |
| タブを右に移動 | `Ctrl+Shift+Right` |

**タブのカスタマイズ:**
- タブを右クリック → 名前の変更、色の変更が可能
- プロジェクトごとに色分けしておくと視認性が上がる

### 4.2 ペイン（画面分割）操作

| 操作 | ショートカット |
|---|---|
| 右にペインを分割 | `Ctrl+Shift+D` |
| 下にペインを分割 | `Ctrl+Shift+E` |
| ペイン間の移動（上下左右） | `Ctrl+Alt+矢印キー` |
| 現在のペインを最大化/元に戻す | `Ctrl+Shift+Enter` |
| ペインを閉じる | `Ctrl+Shift+W`（タブ内最後のペインならタブごと閉じる） |

### 4.3 実用的な使い方の例

**フロントエンド開発の場合:**
```
タブ1「Dev Server」
  ├── 左ペイン: npm run dev（開発サーバー稼働中）
  └── 右ペイン: 作業用シェル（git操作やファイル操作）

タブ2「Testing」
  └── テスト実行用

タブ3「Database」
  └── psql や prisma studio など
```

## 5. コンテキスト注入（Rules / AGENTS.md）

### 5.1 Rulesとは

**Rules（ルール）** は、エージェントに対して永続的なコンテキストを提供する仕組み。プロジェクトの技術スタック、コーディング規約、よくある作業パターンなどを事前に伝えておくことで、エージェントの回答精度が大幅に向上する。

### 5.2 ルールの種類

| 種類 | 設定場所 | 適用範囲 |
|---|---|---|
| **グローバルルール** | Warp Drive > Personal > Rules | すべてのプロジェクトに適用 |
| **プロジェクトルール** | リポジトリルートの `AGENTS.md` | そのプロジェクト内のみ |
| **ディレクトリルール** | 任意のディレクトリの `AGENTS.md` | そのディレクトリ以下で適用 |

### 5.3 AGENTS.md（旧 WARP.md）の書き方

プロジェクトのルートディレクトリに `AGENTS.md` というファイルを作成する（ファイル名は**大文字**で記述すること）。

```markdown
# プロジェクト: My Web App

## 技術スタック
- フロントエンド: React 19 + TypeScript + Vite
- バックエンド: Python 3.12 + FastAPI
- データベース: PostgreSQL 16
- ORM: SQLAlchemy 2.0
- パッケージマネージャ: pnpm（フロント）、uv（バック）

## コーディング規約
- TypeScriptでは関数コンポーネントのみ使用（クラスコンポーネント禁止）
- Python側はtype hintsを必ず付ける
- テストはpytest（バック）とVitest（フロント）を使用
- コミットメッセージはConventional Commitsに従う

## ディレクトリ構成
- `frontend/` - Reactアプリケーション
- `backend/` - FastAPI サーバー
- `infrastructure/` - Terraform / Docker設定
- `docs/` - ドキュメント

## よくある作業
- 開発サーバー起動: `cd frontend && pnpm dev`（フロント）、`cd backend && uvicorn main:app --reload`（バック）
- マイグレーション: `cd backend && alembic upgrade head`
- テスト: `cd frontend && pnpm test` / `cd backend && pytest`

## 注意事項
- `.env` ファイルは絶対にコミットしないこと
- API キーは環境変数から読み取ること
- main ブランチへの直接 push は禁止
```

### 5.4 対応しているルールファイル形式

Warpは以下のファイルを自動認識する（他のAIツールとの互換性あり）:

- `AGENTS.md`（Warp推奨のデフォルト）
- `WARP.md`（旧形式、引き続きサポート）
- `CLAUDE.md`（Claude Code互換）
- `.cursorrules`（Cursor互換）
- `AGENT.md`
- `GEMINI.md`
- `.clinerules`
- `.windsurfrules`
- `.github/copilot-instructions.md`（GitHub Copilot互換）

すでに他のAIツール用にルールファイルを作っている場合、そのまま流用できる。

### 5.5 MCP（Model Context Protocol）サーバーの設定

MCPサーバーを接続することで、エージェントに外部ツールやデータソースへのアクセスを追加できる。

**設定方法1: 起動コマンド方式**
```json
{
  "mcpServers": {
    "my-server": {
      "command": "npx",
      "args": ["-y", "@my-org/mcp-server"],
      "env": {
        "API_KEY": "your-api-key"
      },
      "working_directory": "/path/to/project"
    }
  }
}
```

**設定方法2: SSE（Server-Sent Events）方式**
```json
{
  "mcpServers": {
    "remote-server": {
      "url": "https://my-mcp-server.example.com/sse",
      "headers": {
        "Authorization": "Bearer your-token"
      }
    }
  }
}
```

複数のMCPサーバーを同時に設定でき、`mcpServers` オブジェクト内にキー名を分けて記述すれば自動的にすべて接続される。

## 6. エージェントモード

### 6.1 エージェントモードへの切り替え

| 方法 | 操作 | 備考 |
|---|---|---|
| **`/agent`** | ターミナルの入力欄に `/agent` と入力 | **最も確実。新しい会話ビューが開く** |
| `/new` | ターミナルの入力欄に `/new` と入力 | `/agent` と同様の動作 |
| ショートカット | `Ctrl+I`（デフォルト） | Windows環境ではコンフリクトする場合あり（下記参照） |
| 自動検出 | 入力欄に自然言語を打つと自動切り替え | 設定で有効化が必要 |
| ボタン | メニューバーの ✨ アイコン | エージェントモードの新しいペインが開く |

**注意: `Ctrl+I` がWindows環境で反応しない場合**

Windows環境では `Ctrl+I` がPowerShellや他のツールのキーバインドとコンフリクト（競合）して反応しないことがある。その場合の対処法:

1. **`/agent` を使う（推奨）** — ショートカットキーに依存しないので確実に動く
2. **Settings > Keyboard Shortcuts** で「Toggle Agent Mode」を検索し、空いているキーに再割り当てする

**重要: エージェントモードに切り替えずに自然言語を入力すると、PowerShellがそれをコマンドとして解釈しようとしてエラーになる。** 必ずモードを切り替えてから入力すること。

```
× 通常モードで「このファイルを修正して」と入力
  → PowerShellが「このファイルを修正して」をコマンドとして実行しようとしてエラー

○ /agent と入力してエージェントモードに切り替えてから「このファイルを修正して」
  → AIが指示を理解して適切なコマンドを提案
```

### 6.2 基本的な使い方

エージェントモードに入ったら、やりたいことを日本語または英語で入力するだけ。

**例:**
```
> このディレクトリのPythonファイルで、未使用のimportがあるものを探して修正して
```

エージェントは以下のステップで動作する:
1. 入力を解釈し、必要なコマンドを判断
2. コマンドを提示し、**実行前にユーザーの承認を求める**
3. 承認後にコマンドを実行
4. 出力結果を分析し、次のアクションを判断
5. エラーがあれば修正案を提示

### 6.3 実用的なユースケース

**即座の修正（Immediate Fix）:**
コマンドがエラーで失敗した場合、エージェントモードが約2秒で3つの修正候補をランク付きで提示する。

**マルチステップの作業:**
```
> Dockerfileを作って、Node.js 20ベースで、pnpmを使って依存関係インストールして、
  本番ビルドを実行するマルチステージビルドにして
```
→ エージェントがDockerfile作成 → ビルドテスト → エラーがあれば修正、という流れを自律的に実行

**デバッグ:**
```
> npm run build でエラーが出る。原因を調べて修正して
```
→ ビルド実行 → エラー内容を分析 → 修正を提案・適用

### 6.4 Full Terminal Use（インタラクティブツールとの連携）

すでに起動中の対話型ツール内でもエージェントを呼び出せる。

**対応例:**
- データベースシェル（`psql`、`mysql`）
- デバッガー（`gdb`、`pdb`）
- 開発サーバー（`npm run dev` の出力を監視）
- テキストエディタ（`vim`、`nano`）

**使い方:**
1. 対話型ツールを起動した状態で `Ctrl+I`
2. またはフッターの「Use Agent」をクリック
3. エージェントがそのツールのコンテキストを理解して操作を支援

### 6.5 スラッシュコマンド

エージェントモードの入力欄で `/` を入力すると、利用可能なスラッシュコマンドの一覧が表示される。

| コマンド例 | 説明 |
|---|---|
| `/explain` | 直前のコマンドやエラーの説明 |
| `/fix` | エラーの修正を提案 |
| `/test` | テストコードの生成 |

スラッシュコマンドはカスタマイズ可能で、Warp Driveにプロンプトとして保存しておけば `/` メニューから呼び出せる。

### 6.6 Conversation（会話スレッド）の管理

#### タブ（Tab）とConversationの違い

WarpにおけるTabとConversationはまったく別のレイヤーにある概念である。

| 概念 | 説明 | スコープ |
|---|---|---|
| **Tab（タブ）** | ターミナルの「入れ物」。独立したシェルセッションが動く物理的な場所 | 1つのウィンドウ内 |
| **Conversation（会話）** | エージェントとのやりとりの「スレッド」。**特定のタブに紐づかず、ウィンドウ全体を横断して管理される** | ウィンドウ/ワークスペース全体 |

Conversationはチャットアプリのスレッドに近い。タブ1で始めた会話をタブ2から呼び出して続けることもできるし、会話メニューからいつでも過去の会話に戻れる。

```
┌─ ウィンドウ ────────────────────────────────────────┐
│                                                     │
│  [タブ1: Frontend]  [タブ2: Backend]  [タブ3: DB]     │
│                                                     │
│  ── Conversations（ウィンドウ全体で共有） ──           │
│  │                                                  │
│  ├─ Conversation A: 「Dockerの設定を手伝って」         │
│  ├─ Conversation B: 「テストを書いて」                 │
│  └─ Conversation C: 「このエラーを直して」             │
│                                                     │
│  ※ どのタブからでも会話メニューで切り替え可能          │
│  ※ 過去の会話を選ぶと新しいタブやペインに復元される    │
└─────────────────────────────────────────────────────┘
```

#### Conversationの復元と同期

| 機能 | 説明 |
|---|---|
| **ローカル復元** | 過去の会話を会話メニューから選ぶと、新しいタブまたは現在のペインに復元される |
| **Cloud-synced Conversations** | 設定で有効化すると、会話履歴がクラウドに保存され、**別のデバイスからも会話を復元**できる。チームメンバーとの共有も可能 |

#### Pair Mode（ペアモード）

エージェントモードに入ると、デフォルトで **Pair Mode** になる。Pair Modeでは直前の会話の続きとしてフォローアップの質問やタスクを依頼できる。エージェントは過去のやりとりのコンテキストを保持しているので、関連する質問なら同じConversation内で続けた方が精度が高い。

#### Conversationの操作（Windows）

| 操作 | 方法 |
|---|---|
| 会話メニューを開く | `Ctrl+Shift+Y` |
| 新しいConversationを開始 | 会話メニュー → 「New Conversation」、または `/new` コマンド |
| 過去のConversationに戻る | 会話メニューから選択、または入力欄のピンクの会話チップをクリック |
| Conversationを別ペインで開く | 会話メニューで右クリック → 「Fork in new pane」 |
| Conversationを削除 | 会話メニューで右クリック → 「Delete」 |

**制限事項: Conversationのタイトルは変更できない（2026年3月時点）。** タイトルは最初の質問内容から自動生成され、手動でのリネーム機能は提供されていない。そのため、後から見返したときに分かりやすくしたい場合は、**最初の質問を具体的に書く**のがコツ。

```
× 「教えて」「これ何？」
  → タイトルが曖昧になり、後で何の会話か分からなくなる

○ 「ContainerShippingのDocker環境を構築したい」
  → タイトルから内容が一目でわかる
```

#### 使い分けのベストプラクティス

- **同じConversationで続ける場合**: 質問が前の話題と関連しているとき。文脈を活かせるので回答精度が上がる
- **新しいConversationを始める場合**: まったく別の話題に移るとき。無関係なコンテキストが混ざると精度が落ちる
- **注意点**: Conversationが長くなりすぎると動作が遅くなり、回答の質も下がる。別の作業に移るときは新しいConversationを始めるのが推奨
